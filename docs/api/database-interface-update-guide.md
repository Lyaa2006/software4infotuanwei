# 数据库与接口更新说明文档

本文档是说明文档，不直接作为日常变更记录使用。

任何成员或 coding agent 在修改数据库结构、初始化数据、后端接口、前端接口调用后，都应先阅读本文档，再把本次改动填写到维护文档 [database-interface-update-log.md](./database-interface-update-log.md)。

配套参考：

- [本地人大金仓数据库统一规范](../local-kingbase-setup.md)
- [数据库统一建库步骤](../数据库统一建库步骤.md)
- [API 规格说明（初版模板）](./api-spec.md)

## 1. 文档定位

- 本文档：说明文档，负责定义什么时候记、怎么记、按什么格式记
- 维护文档：记录文档，负责逐条沉淀真实发生过的变更

后续协作约定：

1. 先改代码
2. 再阅读本文档确认是否属于必须记录的变更
3. 最后到 [database-interface-update-log.md](./database-interface-update-log.md) 追加一条记录

## 2. 使用目的

这份文档解决 3 个问题：

1. 记录这次改动到底改了哪些数据库对象和接口。
2. 让其他成员知道本地环境需要同步哪些内容。
3. 降低“代码改了但文档没改、库没同步、前后端联调不上”的风险。

## 3. 什么时候必须更新维护文档

出现以下任一情况时，提交代码前必须更新维护文档：

- 新增表、删表、改表名
- 新增字段、删除字段、修改字段类型或约束
- 新增初始化数据或修改关键初始化数据
- 新增接口
- 修改接口路径、方法、请求参数、响应字段、错误码
- 修改前端调用方式，导致联调参数或返回值要求变化
- 修改登录、权限、RBAC、账号初始化逻辑

如果只是下面这些情况，可以不写维护文档：

- 纯文案修改
- 不影响请求/响应契约的内部重构
- 不影响数据库对象的代码整理
- 注释、排版、命名清理

## 4. 更新规则

### 4.1 一次功能改动至少更新两处

- 在 [database-interface-update-log.md](./database-interface-update-log.md) 中追加一条
- 如果接口契约发生变化，同步更新 [api-spec.md](./api-spec.md) 或对应接口说明

### 4.2 数据库改动必须写清楚

至少写清：

- 改了哪张表
- 改了什么字段
- 是否兼容旧数据
- 其他人本地是否需要重新建表、补 SQL、导入数据

### 4.3 接口改动必须写清楚

至少写清：

- 接口路径和方法
- 请求参数变化
- 响应字段变化
- 前端是否需要同步修改

### 4.4 不允许只写“优化”“调整”

描述必须具体到结构层面，例如：

- 正确：`party_students` 新增 `tags` 字段，类型 `JSONB NOT NULL DEFAULT '[]'::jsonb`
- 错误：优化党员信息表

## 5. coding agent 使用流程

如果后续让 coding agent 帮你补维护记录，按这个流程执行：

1. 让 agent 先阅读本文档
2. 再让 agent 检查本次代码改动是否涉及数据库或接口契约
3. 如果涉及，让 agent 到 [database-interface-update-log.md](./database-interface-update-log.md) 追加一条新记录
4. 如果涉及公共接口，再同步更新 [api-spec.md](./api-spec.md)
5. 合并前检查维护文档里的“本地同步步骤”是否足够让其他成员照做

推荐直接给 agent 的指令模板：

```text
先阅读 docs/api/database-interface-update-guide.md。
根据本次代码改动，判断是否涉及数据库结构、初始化数据、接口契约或前端调用变更。
如果涉及，请按说明文档的规范，把本次变更追加到 docs/api/database-interface-update-log.md。
如果接口契约发生变化，再同步更新 docs/api/api-spec.md。
```

| 日期 | 模块 | 改动类型 | 摘要 | 是否需要同步数据库 | 是否需要同步前端 | 负责人 | 关联提交/PR |
| --- | --- | --- | --- | --- | --- | --- | --- |
| YYYY-MM-DD | 例如：党建流程 / 学业预警 / 提醒通知 | `db` / `api` / `db+api` / `seed` | 一句话概括本次变化 | 是/否 | 是/否 | 张三 | `commit id` 或 `PR #xx` |

## 8. 推荐填写示例

下面给一个简化示例，后续成员可以直接参考措辞。

## 更新记录：2026-05-22 - 提醒通知 - 学生标签字段联动

### 1. 基本信息

| 字段 | 内容 |
| --- | --- |
| 模块 | 提醒通知 |
| 改动类型 | `db+api` |
| 负责人 | 张三 |
| 关联提交/PR | `abc1234` |
| 影响范围 | `server` / `webpage` / `miniprogram` |
| 是否破坏兼容性 | 否 |
| 是否要求其他成员同步操作 | 是 |

### 2. 改动摘要

- 本次改了什么：`party_students` 表新增 `tags` 字段，提醒通知相关接口支持按标签发送。
- 改动原因：支持管理员按学生标签筛选通知对象。
- 主要风险：旧库如果没有该字段，提醒通知和部分活动接口会报错。

### 3. 数据库变更

#### 3.1 涉及表

| 表名 | 操作类型 | 说明 |
| --- | --- | --- |
| `party_students` | 修改 | 新增学生标签字段 |

#### 3.2 字段变更明细

| 表名 | 字段名 | 操作类型 | 变更前 | 变更后 | 是否必填 | 默认值 | 备注 |
| --- | --- | --- | --- | --- | --- | --- | --- |
| `party_students` | `tags` | 新增 | 无 | `JSONB` | 是 | `'[]'::jsonb` | 用于标签筛选 |

#### 3.3 SQL / 建表同步要求

```sql
ALTER TABLE party_students
ADD COLUMN tags JSONB NOT NULL DEFAULT '[]'::jsonb;
```

#### 3.4 初始化数据变更

- 是否涉及初始化数据：否
- 涉及表：无
- 变更内容：无
- 是否需要重新导入旧数据：否

### 4. 接口变更

| 接口名称 | 方法 | 路径 | 操作类型 | 调用方 |

| 学生列表 | `GET` | `/api/reminder/admin/students` | 修改 | `webpage` / `miniprogram` |
| 保存学生标签 | `PUT` | `/api/reminder/admin/students/:accountId/tags` | 新增 | `webpage` / `miniprogram` |

#### 4.3 响应字段变化

| 接口 | 字段名 | 变更前 | 变更后 | 影响说明 |
| --- | --- | --- | --- | --- |
| `/api/reminder/admin/students` | `items[].tags` | 无 | `string[]` | 前端列表新增标签展示 |

### 5. 前端联调影响

- `webpage` 是否需要修改：是
- `miniprogram` 是否需要修改：是

1. `git pull`
2. 执行新增字段 SQL，或直接启动后端让其自动补字段
3. 重启后端

## 9. 团队约定

- 每次新增接口，必须同步补“涉及接口”部分。
- 每次改表结构，必须同步补“字段变更明细”部分。
- 如果改动会影响别人本地库，必须写“本地同步步骤”。
- 如果只是修文案、不影响契约和数据库，可以不记入本文档。
- 维护文档按时间倒序追加，新记录放在最前面。
