# 第一次讨论会议议程模板

本文档用于项目第一次正式讨论会，目标是把“先做什么、谁负责什么、哪些契约先定下来”一次说清，避免后续各自开工后反复返工。

## 1. 会议目标

- 明确第一轮开发只做哪些内容，不做哪些内容
- 明确四位成员的 owner 和写入边界
- 明确第一批接口讨论范围和负责模块
- 明确第一批角色、权限、数据范围讨论范围
- 明确数据库设计和迁移脚本由谁主导
- 明确开发前必须先补的文档和后续协作规则

## 2. 会前准备

参会前每个人先阅读：

- `README.md`
- `docs/team-vibecoding-plan.md`
- `docs/api/api-spec.md`
- `docs/rbac-matrix.md`

会前准备要求：

- 成员 A：整理认证、角色、权限、数据范围的待确认项
- 成员 B：整理党团流程、学业分析、导入导出的待确认项
- 成员 C：整理学生端需要依赖的接口和权限展示需求
- 成员 D：整理管理端需要依赖的接口和权限展示需求

## 3. 会议产出要求

本次会议结束后，至少要产出以下结果：

- 第一轮开发范围清单
- 四位成员的 owner 列表
- 第一批接口清单和接口 owner
- 第一批角色清单、数据范围和权限讨论结论
- 数据库设计负责人与改表规则
- 会后要补充的 issue 或文档清单

## 4. 议程安排

### 4.1 开场与目标对齐

需要确认：

- 这次项目第一轮的目标是什么
- 第一轮只做骨架，还是要做到可演示
- 第一轮哪些模块必须进，哪些模块先不做

本节输出：

- 第一轮开发范围
- 第一轮暂不做内容

建议时长：

- 10 分钟

### 4.2 技术栈与工程骨架

需要确认：

- 前端框架选型
- 后端框架选型
- 数据库与迁移方案
- 是否先补 Docker / 本地启动方案
- 前后端目录是否按现有结构推进

本节输出：

- 前端技术栈结论
- 后端技术栈结论
- 数据库与迁移方式结论
- 是否立即初始化 `frontend/`、`backend/`

建议时长：

- 15 分钟

### 4.3 成员分工与 ownership

需要确认：

- 成员 A、B、C、D 分别负责哪些目录和模块
- 哪些公共文件只能由 owner 改
- 谁负责 `database/migrations`
- 谁负责 `docs/api/`
- 谁负责 `docs/rbac-matrix.md`

本节输出：

- 每个人的负责目录
- 每个人的负责模块
- 公共文件改动规则

建议时长：

- 10 分钟

### 4.4 第一批接口讨论

讨论依据：

- `docs/api/api-spec.md`

需要确认：

- 登录接口
- 获取当前用户信息接口
- 获取当前用户权限接口
- 党团流程查询接口
- 学业分析上传或查询接口
- 导入导出和审计日志是否进入第一轮

每个接口至少先确认：

- 路径
- 方法
- 调用方
- 核心参数
- 返回结构是否先统一
- owner 是谁

本节输出：

- 第一批接口清单
- 每个接口的 owner
- 当前轮次先冻结的接口范围

建议时长：

- 20 分钟

### 4.5 第一批角色与权限讨论

讨论依据：

- `docs/rbac-matrix.md`

需要确认：

- 第一轮保留哪些角色
- 是否支持一人多角色
- 数据范围是否采用 `self / class / college / all`
- 哪些能力必须严格后端鉴权
- 哪些能力只做前端展示控制
- 审批、导出、日志查看权限先怎么定

本节输出：

- 第一批角色清单
- 数据范围定义
- 第一批关键权限点范围
- 前端展示权限和后端真实权限的分工原则

建议时长：

- 20 分钟

### 4.6 数据库设计与变更规则

需要确认：

- 第一轮先设计哪些核心表
- 用户、角色、权限相关表由谁负责
- 业务表由谁负责
- 其他成员如果需要改表，如何提需求
- 迁移脚本命名和提交规则是否现在就定

本节输出：

- 第一批核心表范围
- 数据库 owner
- 数据库变更流程

建议时长：

- 15 分钟

### 4.7 协作与提交规则确认

讨论依据：

- `docs/git-flow-collaboration.md`
- `docs/team-vibecoding-plan.md`

需要确认：

- 是否按当前 `main / develop / feature/* / hotfix/*` 规则执行
- PR 默认主审和专项复审是否按现有建议执行
- 每天同步机制怎么做
- 文档、接口、数据库变更后是否要求同步 issue / PR 说明

本节输出：

- 分支和 PR 规则确认
- 每日同步方式
- 文档同步规则

建议时长：

- 10 分钟

### 4.8 会后任务拆分

需要确认：

- 谁先补 `docs/api/api-spec.md`
- 谁先补 `docs/rbac-matrix.md`
- 谁先初始化前端
- 谁先初始化后端
- 谁先整理数据库设计初稿

本节输出：

- 会后第一批任务列表
- 每项任务负责人
- 截止时间

建议时长：

- 10 分钟

## 6. 主持人控制要求

- 不要在第一次会议里把所有业务细节展开到底
- 先定边界、owner、第一批接口、第一批权限，不要一上来就讨论页面细节
- 有争议但不影响第一轮开工的内容，标记为 `待确认`，会后开 issue
- 会议中讨论到接口或权限时，直接回填到对应文档，不要只停留在口头结论
- 会议结束前必须确认每个人下一步要做什么，避免“都懂了但没人动”

## 7. 当前建议会议顺序

1. 先定第一轮范围
2. 再定技术栈和工程骨架
3. 再定 owner 和目录边界
4. 再定第一批接口
5. 再定第一批角色与权限
6. 再定数据库与迁移规则
7. 最后确认协作机制和会后任务
