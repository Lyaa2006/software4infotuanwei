# 网页端学生个人主页与密码重置功能规划

## 1. 目标

本次仅实现网页端最简版本的学生个人主页与密码重置能力，暂不接入邮箱验证码。目标如下：

1. 学生在登录页点击“忘记密码？”后，可进入重置密码页面。
2. 学生在未登录状态下，输入学号并填写两次新密码后，可直接完成密码重置。
3. 学生登录后，首页顶栏中的“学生 · 学号”身份标识改为可点击按钮，跳转至个人主页。
4. 新增学生个人主页，展示姓名、学号，并提供“重置密码”入口。
5. 学生在已登录状态下，也可从个人主页跳转至重置密码页面完成密码修改。

## 2. 范围

本次仅覆盖网页端：

- `webpage/` 页面与路由
- `server/` 登录与密码重置接口
- `docs/` 规划文档

暂不覆盖：

- 小程序端同步改造
- 邮箱验证码发送
- 管理员重置学生密码
- 通用账户安全增强机制（频控、验证码、二次确认邮件等）

## 3. 现状分析

当前系统登录逻辑具备以下特征：

1. 用户首次登录时，如账号在 `permitted_accounts` 中且 `users` 表内不存在记录，系统会使用本次输入密码创建账户。
2. 系统数据库仅保存 `password_hash` 与 `salt`，不保存明文密码。
3. 当前网页端无“忘记密码”入口，无个人主页页面，也无密码重置接口。

因此，本次最简方案不做“找回原密码”，而是新增“直接重置为新密码”的能力。

## 4. 实现步骤

### 4.1 后端接口

新增密码重置接口：

- 路径：`POST /api/auth/reset-password`
- 参数：
  - `role`
  - `accountId`
  - `newPassword`
  - `confirmPassword`

处理规则：

1. 仅允许 `student` 角色使用该最简重置能力。
2. 学号必须存在于 `permitted_accounts` 且启用。
3. 两次新密码必须一致。
4. 新密码不能为空，并满足基础长度要求。
5. 若用户在 `users` 表中不存在，则按当前系统既有逻辑直接创建学生账户并写入新密码。
6. 若用户已存在，则更新其 `password_hash`、`salt`。

### 4.2 登录页

在网页端登录页增加“忘记密码？”按钮：

1. 仅在当前角色选择为学生时显示。
2. 点击后跳转到重置密码页面。
3. 若当前已填写学号，则携带学号作为页面初始值。

### 4.3 个人主页

新增网页端学生个人主页页面：

展示内容：

1. 姓名
2. 学号
3. 重置密码按钮

数据来源：

- 优先使用当前登录会话中的学号
- 姓名可复用学生党团信息接口返回的数据

### 4.4 首页顶栏入口

将首页顶栏原有“学生 · 学号”身份标识改为可点击按钮：

1. 管理员保持原样或不跳转。
2. 学生点击后跳转至个人主页。

### 4.5 重置密码页面

新增网页端重置密码页面：

页面字段：

1. 学号
2. 新密码
3. 确认新密码

交互规则：

1. 未登录从登录页进入时，允许学生直接填写学号重置。
2. 已登录从个人主页进入时，自动带入当前学生学号并可只修改密码。
3. 提交成功后提示重置成功，并自动跳转回登录页。
4. 若当前处于登录态，成功后应清除本地会话，避免旧状态继续使用。

## 5. 注意事项

### 5.1 安全性说明

本次方案为最简验证版本，存在明显安全边界不足：

1. 未接入邮箱验证码，意味着只要知道学号且该学号在权限名单中，即可直接重置密码。
2. 该能力仅适合课程设计阶段的最小功能验证，不适合直接作为正式生产方案长期使用。

### 5.2 后续增强建议

后续应尽快补充：

1. 邮箱验证码或邮件重置链接
2. 重置频率限制
3. 操作日志记录
4. 更严格的密码强度规则
5. 登录后修改密码时校验旧密码

## 6. 验收点

完成后至少验证以下场景：

1. 学生首次登录仍可按现有逻辑创建账户。
2. 登录页“忘记密码？”可跳转到重置密码页。
3. 学生输入两次一致的新密码后可成功重置。
4. 两次密码不一致时应提示错误。
5. 不存在于权限名单中的学号不得重置。
6. 学生登录后点击首页顶栏身份按钮可进入个人主页。
7. 个人主页点击“重置密码”后可跳转并完成密码修改。
