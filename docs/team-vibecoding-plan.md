# 团队 Vibecoding 分工与协作策略

本文档用于明确四人团队在项目第一轮开发中的任务分工、并行边界和协作规则，目标是减少多人同时使用 vibecoding 时产生的代码冲突、接口不一致、数据库反复变更和 AI 误改公共文件等问题。

## 1. 文档适用范围

本文档适用于当前项目的第一轮并行开发，重点覆盖：

- 第一轮任务拆分
- 每位成员的职责边界
- 推荐功能分支命名
- 团队使用 vibecoding 的重要注意事项
- 当前阶段应执行的管理策略

## 2. 第一轮开发目标

第一轮不追求所有业务一次做完，而是先建立一个可继续扩展的基础骨架。当前阶段优先目标如下：

1. 建立登录、用户、角色、权限的基础能力
2. 建立学生端和管理端的前端基础框架
3. 建立至少一个核心业务模块的后端雏形
4. 建立接口文档、数据库变更和 PR 审查的基本协作机制

第一轮结束时，团队应至少得到以下成果：

- 仓库具备可持续开发的代码骨架
- 团队成员有清晰的目录 ownership
- 第一批接口文档和数据库设计初稿落地
- 学生端和管理端各有一个可运行的页面框架
- 后端完成基础鉴权和至少一个核心业务模块

## 2.1 当前阶段先看哪两份文档

在团队还没开始正式商量接口和权限之前，先统一参考以下两份文档：

- `docs/api/api-spec.md`：先看“当前要先商量的内容”和“第一批接口清单”，用于整理接口讨论范围
- `docs/rbac-matrix.md`：先看“当前要先商量的内容”和角色、资源、范围骨架，用于整理权限讨论范围

当前阶段这两份文档的定位不是最终定稿，而是：

- 帮团队先列清楚要讨论什么
- 帮不同成员知道自己开工前要确认哪些契约
- 帮后续 issue、接口实现、菜单控制和 Review 有统一落点

## 3. 第一轮详细分工

建议四人团队第一轮采用“基础设施 1 人、后端业务 1 人、学生端前端 1 人、管理端前端 1 人”的分工方式。

### 3.1 成员 A：认证、权限与基础设施负责人

负责内容：

- 用户登录基础方案
- 用户表、角色表、权限表初稿
- 鉴权中间件或权限校验基础逻辑
- 操作日志 / 审计日志基础结构
- 环境变量读取、基础配置文件组织方式

建议功能分支：

- `feature/auth-rbac-foundation`

主要交付物：

- 后端鉴权基础代码
- 权限模型初稿
- 日志模型或日志记录方案
- `docs/api/` 中与登录、权限相关的接口说明
- `docs/rbac-matrix.md` 中角色、数据范围、权限点命名初稿
- `database/` 中与用户、角色、权限相关的设计草稿

开始前先看：

- 先看 `docs/rbac-matrix.md` 开头的“当前要先商量的内容”，整理角色、数据范围、权限点的讨论提纲
- 再看 `docs/api/api-spec.md` 开头和第一批接口清单，整理认证和权限相关接口草案

本轮负责重点模块：

- `auth.session`
- `user.profile`
- 角色模型、权限点命名、数据范围定义
- `audit.log` 的权限边界和基础结构

本轮尽量不要负责：

- 学生端页面细节
- 管理端业务页面
- 大量具体业务模块逻辑

容易出错的点：

- 把前端展示权限误当成后端真实权限控制
- 角色、权限、用户三者关系设计过早固定，后续难扩展
- JWT、会话、权限缓存等配置命名不统一
- 审计日志字段设计不完整，后续无法追溯关键操作

容易影响他人的点：

- 一旦修改登录接口、当前用户信息接口、权限字段命名，会直接影响前后端所有调用方
- 一旦修改全局配置读取方式，会影响整个后端启动和部署方式
- 一旦调整角色模型或权限判断逻辑，会影响管理端、学生端和业务接口的联调

### 3.2 成员 B：后端业务模块负责人

负责内容：

- 第一批核心业务模块后端逻辑
- 党团流程相关接口雏形
- 学业分析相关接口雏形
- 文件导入导出后端基础结构
- 与数据库表结构对应的业务模型初稿

建议功能分支：

- `feature/backend-party-process`
- 或 `feature/backend-core-modules`

主要交付物：

- 至少一个核心业务模块的接口雏形
- 基础的数据模型或服务层结构
- 第一批业务接口文档
- `docs/rbac-matrix.md` 中业务资源与操作项补充
- 如需改表，配套数据库设计说明

开始前先看：

- 先看 `docs/api/api-spec.md` 的第一批接口清单，确认自己负责先起草哪些业务接口
- 再看 `docs/rbac-matrix.md` 的资源与操作清单，补充党团流程、学业分析、导入导出相关资源定义

本轮负责重点模块：

- `party.process`
- `academic.analysis`
- `import.task`
- `export.task`

本轮尽量不要负责：

- 登录鉴权总设计
- 学生端和管理端页面框架

容易出错的点：

- 接口命名、返回结构和字段语义不稳定，导致前端频繁返工
- 在业务代码中直接写死权限判断，和成员 A 的统一权限方案冲突
- 导入导出、文件处理逻辑与数据库结构耦合过深
- 只考虑功能跑通，没有明确异常返回和错误提示格式

容易影响他人的点：

- 一旦修改接口路径、请求参数或返回字段，会直接影响成员 C、D 的前端对接
- 一旦修改业务表结构，会影响数据库迁移、测试数据和后续联调
- 一旦改动公共 service 或通用工具层，可能影响其他后端模块

### 3.3 成员 C：学生端前端负责人

负责内容：

- 学生端项目初始化
- 学生端路由和基础布局
- 首页、个人信息、基础导航骨架
- 与学生使用场景相关的页面占位结构

建议功能分支：

- `feature/frontend-student-core`

主要交付物：

- 学生端前端项目骨架
- 基础页面与导航结构
- API 调用层占位代码
- 与后端对接所需的接口字段说明反馈

开始前先看：

- 在开始写页面骨架和请求层之前，先看 `docs/api/api-spec.md` 的第一批接口清单和对应草案
- 在开始写菜单显示、按钮显示、个人数据页面之前，先看 `docs/rbac-matrix.md`，确认学生端只按已讨论范围做展示控制

本轮负责重点模块：

- 学生端登录后的首页、个人信息、导航骨架
- 学生端对 `auth.session`、`user.profile`、`party.process`、`academic.analysis` 的调用占位
- 学生端权限展示反馈，不负责定义最终权限规则

本轮尽量不要负责：

- 管理端页面
- 全局权限逻辑的最终定义

容易出错的点：

- 在接口未冻结时先写死字段，后续大面积返工
- 为了让页面先跑通，私自写出与接口约定不一致的 mock 数据结构
- 让 AI 顺手改动全局路由、公共请求层或布局文件
- 页面骨架阶段就过早引入复杂状态管理，增加后续维护成本

容易影响他人的点：

- 一旦修改全局请求封装、登录态存储方式，会影响管理端前端
- 一旦修改学生端公共组件命名或布局结构，后续多人协作会混乱
- 如果把接口字段理解错并固化到页面逻辑中，联调时会放大后端变更成本

### 3.4 成员 D：管理端前端负责人

负责内容：

- 管理端项目初始化
- 管理后台基础布局、菜单、导航
- 审批页、通知页、导入页等核心管理页骨架
- 预留日志查看与数据管理入口

建议功能分支：

- `feature/frontend-admin-core`

主要交付物：

- 管理端前端项目骨架
- 管理后台页面框架
- 与审批、导入导出相关的页面占位结构
- 与后端对接的接口需求说明

开始前先看：

- 在开始搭管理端菜单、审批页、导入导出页之前，先看 `docs/api/api-spec.md` 的第一批接口清单
- 在开始写菜单权限、按钮权限、日志入口和审批入口之前，先看 `docs/rbac-matrix.md`，避免先把权限写死

本轮负责重点模块：

- 管理端布局、菜单、路由骨架
- 管理端对 `party.process`、`academic.analysis`、`import.task`、`export.task`、`audit.log` 的页面占位
- 管理端权限展示需求反馈，不负责最终 RBAC 定义

本轮尽量不要负责：

- 数据库结构设计
- 登录鉴权基础能力总设计

容易出错的点：

- 审批页、导入页、后台菜单结构通常依赖权限和接口，若过早写死，后续很难调整
- 管理端页面字段通常更多、更复杂，容易与后端约定不一致
- AI 容易顺手把管理端页面和公共布局、全局菜单一起重构
- 导入导出页面容易忽略失败提示、权限提示和空状态处理

容易影响他人的点：

- 一旦修改管理端全局布局、菜单配置或路由守卫，会影响整套后台页面
- 一旦误解审批状态、导出权限、日志字段，会迫使后端或权限层反复调整
- 如果直接改公共请求层或登录态逻辑，会影响成员 C 的学生端实现

## 4. 第一轮并行边界

为了降低冲突，第一轮必须控制每个人的写入范围。

### 4.1 推荐 ownership

- `backend/auth/`、`backend/common/`、`backend/config/`：成员 A 优先负责
- `backend/modules/party/`、`backend/modules/academic/`、`backend/modules/import/`：成员 B 优先负责
- `frontend/student/` 或学生端相关目录：成员 C 优先负责
- `frontend/admin/` 或管理端相关目录：成员 D 优先负责
- `database/`、`database/migrations/`：优先由 A、B 协同，其他成员不要直接改
- `docs/api/`：接口归属人先写，相关调用方补充，不要多人同时大改同一文档

### 4.2 当前阶段禁止多人同时频繁改动的区域

- 登录鉴权相关公共文件
- 权限判断公共逻辑
- 数据库迁移脚本
- 全局路由文件
- 全局布局文件
- 统一请求封装文件
- 统一环境配置文件

这些区域一旦多人同时让 AI 改，很容易出现结构级冲突。

## 5. 第一轮建议开发顺序

建议按以下顺序推进：

1. 先由成员 A、B 确定基础数据模型、角色定义和第一批接口草案
2. 再由成员 C、D 按已经冻结的接口草案搭建前端骨架
3. 后端业务接口和前端页面同时推进，但接口命名与返回格式不得随意变动
4. 每天同步一次接口变化、数据库变化和公共文件变化

第一轮不要一开始就四个人同时自由扩展需求，否则冲突会快速放大。

## 6. 团队使用 Vibecoding 的重要注意事项

### 6.1 先定边界，再让 AI 写代码

AI 很擅长补代码，但不擅长替团队自动维持边界。开始开发前必须先明确：

- 谁负责哪个目录
- 谁可以改哪些公共文件
- 当前接口契约是什么
- 当前数据库结构是否已冻结

如果边界不清楚，AI 会把本来独立的任务写成相互耦合的大改动。

### 6.2 每次只让 AI 处理一个明确任务

不要给 AI 一个过大的模糊任务，例如：

- “把整个后台都做了”
- “顺便帮我把权限和页面都完善一下”

更好的方式是：

- “只生成学生端首页布局，不要改路由之外的文件”
- “只补充党团流程查询接口，不修改数据库结构”
- “只为当前页面接入已存在接口，不要重构公共请求层”

任务越边界清晰，冲突越少。

### 6.3 提交前必须人工检查 diff

vibecoding 最危险的地方之一，是 AI 容易顺手改掉与你任务无关的文件。提交前必须检查：

- 是否改到了不属于自己职责范围的文件
- 是否出现无意义格式化大改
- 是否修改了公共命名、全局路由或请求封装
- 是否误删了他人的结构

不要把“AI 自动改了，但我没细看”带进仓库。

### 6.4 公共文件不要随便让 AI 重写

以下文件或区域，尽量只由负责人改：

- 权限相关公共模块
- 数据库迁移文件
- 全局配置文件
- 全局接口封装
- 顶层布局和路由

AI 对这些文件的“顺手优化”通常比普通业务文件更容易引发连锁问题。

### 6.5 不要一边写一边频繁改契约

如果今天后端定义返回字段为：

- `userId`
- `roleCode`
- `permissions`

那前端就按这一版先接。除非团队同步确认，否则不要在 vibecoding 过程中随意改成另一套字段名。

AI 很容易在不同上下文里生成两套命名风格，导致联调困难。

### 6.6 小 PR 比大 PR 更重要

使用 vibecoding 时，更要坚持小步提交、小 PR。原因是：

- AI 生成的代码量通常偏大
- 一次改动越多，越难 Review
- 冲突定位会更困难
- 出问题时更难回滚

建议：

- 一个 PR 只解决一个明确问题
- 一个 PR 不要同时改数据库、接口、前端页面和文档四大块

## 7. 当前应执行的管理策略

### 7.1 建立目录 ownership

当前团队应尽快补一份简单 ownership 表，至少写清：

- 谁负责 `backend/auth`
- 谁负责 `backend/modules`
- 谁负责 `frontend/student`
- 谁负责 `frontend/admin`
- 谁负责 `database/migrations`
- 谁负责 `docs/api`

没有 ownership，vibecoding 的风险会非常高。

### 7.2 先冻结第一批接口契约

当前应优先补一份第一批接口清单，至少包括：

- 登录接口
- 获取当前用户信息接口
- 获取角色 / 权限接口
- 党团流程查询接口
- 学业分析上传或查询接口

接口契约先冻结，再并行开发。

### 7.2.1 当前阶段这两份文档怎么用

在正式商量之前，先这样使用：

1. 成员 A、B 先分别阅读 `docs/api/api-spec.md` 和 `docs/rbac-matrix.md` 开头的“当前要先商量的内容”
2. A 先补认证、权限、角色、数据范围相关待确认项，B 先补业务模块资源和接口待确认项
3. 成员 C、D 在搭前端骨架前，先按这两份文档确认自己依赖哪些接口和权限，不自行补最终规则
4. 团队开会时，先用这两份文档逐条讨论，而不是临时口头散聊
5. 讨论完成后，再回填 owner、issue、状态和联调备注

这两份文档在第一轮中的分工建议：

- `docs/api/api-spec.md`：由接口 owner 先写，调用方补字段和联调反馈
- `docs/rbac-matrix.md`：由成员 A 主导，成员 B 按业务资源补充，成员 C、D 只补展示和调用层反馈

### 7.3 数据库变更集中管理

数据库结构和迁移脚本建议由少数成员集中维护，当前建议：

- 成员 A 负责权限相关表
- 成员 B 负责业务模块相关表
- 其他成员如需改表，先提需求，不直接提交迁移脚本

### 7.4 公共文件修改先同步

如果要改以下内容，必须先在团队里同步：

- 新增环境变量
- 修改接口返回格式
- 修改数据库核心表字段
- 修改全局路由结构
- 修改统一请求封装

### 7.5 每日同步机制

建议每天至少一次简短同步，每人只说三件事：

1. 今天改了什么
2. 改到了哪些公共文件、接口或表结构
3. 哪些部分别人暂时不要碰

这个机制对 vibecoding 团队非常关键。

### 7.6 PR 审查重点

当前每个 PR 审查时，重点看：

- 是否越界修改了他人负责区域
- 是否改动过大
- 是否包含未沟通的接口变更
- 是否包含未说明的数据库变更
- 是否误改公共文件
- 是否把 AI 生成的无关代码一起提交了

### 7.7 PR 审查责任分配

当前阶段建议采用“默认主审 + 按领域专项复审”的方式，而不是把所有 PR 审核压力都压给同一个人。

默认主审：

- 成员 C 作为默认主审人

原因：

- 第一轮里，学生端前端任务相对更收敛，冲突面和全局耦合度低于认证、后端核心业务和管理端后台
- 更适合承担大多数 PR 的流程性检查，例如改动边界、无关文件、PR 说明完整性等

默认主审主要检查：

- 是否改到了不该改的文件
- 是否混入了 AI 生成的无关改动
- PR 标题和说明是否完整
- 是否按小 PR 原则提交
- 是否存在明显的目录越界

专项复审规则：

- 涉及登录、鉴权、角色、权限、日志审计：必须由成员 A 复审
- 涉及后端业务接口、数据库表结构、导入导出逻辑：必须由成员 B 复审
- 涉及管理端布局、审批流前端、导入导出页面：必须由成员 D 复审

执行方式：

- 普通前端页面 PR：成员 C 主审，必要时相关 owner 补充审核
- 权限或数据库相关 PR：成员 C 做流程检查，A 或 B 做技术复审
- 涉及多个领域的 PR：拆小；若不能拆，则至少让对应 owner 都看过再合并

## 8. 当前推荐的第一轮分支规划

建议第一轮优先使用以下分支：

- `feature/auth-rbac-foundation`
- `feature/backend-party-process`
- `feature/frontend-student-core`
- `feature/frontend-admin-core`

如需要补文档或接口说明，可在对应功能分支内一并提交，不建议再开大量零碎分支。

## 9. 当前阶段不建议的做法

- 四个人同时修改数据库迁移脚本
- 四个人都能改全局配置和公共请求层
- 未确定接口就先各自让 AI 生成整套前后端代码
- 一个 PR 里同时混入多个模块改动
- 不看 diff 直接提交 AI 生成结果
- 遇到冲突后直接覆盖他人代码

## 10. 建议的执行方式

当前最务实的推进方式是：

1. 先确认四位成员对应的第一轮 owner
2. 先冻结第一批接口和数据库设计范围
3. 每人从 `develop` 拉自己的 `feature/*` 分支
4. 只在自己负责区域内使用 vibecoding
5. 小步提交，小 PR 合并回 `develop`
6. 第一轮完成后再统一集成、联调、修复
